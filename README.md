# MALAK Sentinel — AI-Generated Suricata Rules

Community-reviewed Suricata detection rules generated by the **MALAK Sentinel** threat intelligence pipeline.

## What is this?

MALAK Sentinel is an automated CVE-to-rule pipeline:

```
CISA KEV / NVD → A-Hunt (Kimi/Claude LLM draft) → A-Forge (qwen2.5:7b contextualise) → A-Test (suricata -T validation) → Production
```

Every rule in this repository:
- Was drafted by an LLM based on public CVE data only
- Passed `suricata -T` syntax validation on **Suricata 8.0.4**
- Has been running in production on passive TAP sensors

## Feedback welcome

If you spot an issue or have an improvement:

- **Twitter/X**: post your suggested rule with `#Suricata #MALAK` and tag us
- **GitHub**: open an issue or PR against this repo
- **Paste your rule** into our [AI Rule Export page](https://malak.malakoff.com.my:8443/rule-gist) — it will be syntax-checked via `suricata -T` and queued for analyst review before deployment

## Conventions

- **SID range**: `1000000–1999999` (MALAK-reserved)
- **Sticky buffers**: Suricata 7/8 style (`http.uri`, `http.request_body`, `dns.query`, etc.)
- **`detection_filter`**: used on broad PCRE rules to suppress FP noise
- **`malak_origin a_forge`**: rule body generated by A-Forge (qwen2.5:7b)
- **`source: a_forge`**: rule went through the full AI pipeline

## Contributing

Suggested rules submitted via the import page go through:
1. `suricata -T` syntax check — rejected immediately if it fails
2. Analyst review in the MALAK Rule Candidates queue
3. 48-hour stabilisation window in staging before production promotion
